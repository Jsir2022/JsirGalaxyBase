# CustomMod Architecture

日期：2026-03-28

这份文档定义 `CustomMod` 的唯一正式代码架构。
后续新增功能、重构和评审都以这里为准，不再继续沿用早期示例式、扁平式写法。

## 目标

`CustomMod` 的目标不是做一个零散功能合集，而是做一个：

- `一个 mod jar`
- `一个制度核心`
- `多个内部模块`
- `可逐步接入中心化存储和外部网关`

它要同时承载：

- 职业
- 经济
- 贡献度 / 声望
- 福利
- 公共订单 / 公共工程
- 群组服同步
- 跨服传送
- 后续玩法能力模块
  - 例如：`连锁挖掘`
  - 例如：`共享背包`
  - 例如：`市场终端`

## 核心原则

### 1. 模块化单体

- 交付物保持为一个 `mod jar`
- 代码按领域和能力拆成内部模块
- 不拆成多个强耦合小模组

### 2. 核心制度与玩法能力分层

- `制度核心模块`
  - 负责长期稳定的服务器制度规则
  - 包括职业、经济、声望、订单、跨服状态
- `能力模块`
  - 负责可扩展的玩法和工具能力
  - 例如连锁挖掘、共享背包、调试工具

能力模块不能直接侵入制度核心内部状态。
如需联动，必须通过明确接口或内部事件完成。

### 3. 服务端权威

- 客户端只能做输入、展示和界面
- 规则判定、状态变更、掉落处理、账本变更必须由服务端主导
- 任何可能影响经济、权限、同步一致性的能力，都不能以客户端为真源

### 4. 持久化可替换

- 单服阶段允许落本地存储
- 跨服阶段切到中心化存储
- 领域规则不能依赖某个具体存储实现

### 5. 新功能默认按模块接入

后续新增功能，默认按模块接入，而不是继续往主类、代理类、命令类里堆逻辑。

## 分层结构

### 一、Bootstrap 层

负责：

- Forge 生命周期接入
- 配置加载
- 模块装配
- 生命周期广播
- 命令注册

对应包：

- `com.u24game.custommod.bootstrap`

### 二、Module 层

负责：

- 定义统一模块抽象
- 管理模块生命周期
- 提供模块清单和模块元数据

对应包：

- `com.u24game.custommod.module`

### 三、Institution Core 层

负责：

- 职业
- 经济
- 声望 / 贡献度
- 公共订单
- 公共工程
- 群组服同步核心状态

对应包：

- `com.u24game.custommod.modules.core`

### 四、Capability 层

负责：

- 连锁挖掘
- 共享背包
- 市场终端
- 其他玩法增强或生产力功能

对应包：

- `com.u24game.custommod.modules.capability`

### 五、Diagnostics / Tooling 层

负责：

- 调试导出
- 运维辅助
- 开发期观测工具

对应包：

- `com.u24game.custommod.modules.diagnostics`

## 模块设计约束

每个模块都必须有：

- 稳定的 `id`
- 可读的 `displayName`
- 明确的 `category`
- 独立的生命周期方法

模块可以做：

- 注册事件监听器
- 注册命令
- 注册网络消息
- 管理自己的配置和持久化适配器

模块不应做：

- 直接操作其他模块的内部字段
- 在任意静态工具类里共享可变状态
- 把核心规则散落到 GUI、命令和事件监听器里

## 关于连锁挖掘的架构位置

连锁挖掘属于：

- `能力模块`

不属于：

- `制度核心模块`

原因：

- 它是玩法能力，不是制度真源
- 它应独立管理自己的输入、状态和限制
- 它未来可以和职业、权限、领地规则联动
- 但联动必须通过清晰接口，而不是直接写死在核心经济或职业逻辑里

连锁挖掘模块的长期约束：

- 服务端权威
- 有性能上限
- 有方块和维度白名单 / 黑名单
- 有权限 / 职业 / 区域联动入口
- 有审计与调试留痕能力

## 禁止继续使用的旧写法

以下写法不再作为项目规范：

- `Hello World` 风格的验证命令作为主功能骨架
- 把所有逻辑堆在根包下
- 用单个扁平 `Config` 类承载全部未来配置意图
- 用零散静态管理器直接横跨整个项目共享状态
- 先写功能、后补模块边界

## 新功能接入流程

新增功能时按下面顺序做：

1. 先判断它属于 `制度核心模块` 还是 `能力模块`
2. 再确定它是否需要本地存储、共享存储或外部真源适配
3. 给它建立独立模块类
4. 只通过模块生命周期接入事件、命令和网络
5. 再补 GUI、配置和持久化

如果一个功能既影响玩法，又影响制度，例如：

- 职业专属挖掘加成
- 公共矿区连锁限制
- 连锁挖掘产生贡献度

正确做法是：

- 能力模块负责采集事实
- 制度核心负责解释规则和结算结果

## 当前实施结论

从本次重构开始，`CustomMod` 统一采用：

- `模块化单体`
- `制度核心 + 能力模块`
- `服务端权威`
- `可替换持久化`

后续代码应基于这个骨架逐步填充业务，而不是继续保留早期样板式实现。
