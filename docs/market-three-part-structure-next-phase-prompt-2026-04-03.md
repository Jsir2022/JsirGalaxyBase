# 市场三分结构下一阶段 Prompt

你现在接手的不是旧的单一路线市场收尾，也不是继续把 MARKET 页往“单一商品撮合页”上堆功能。

当前正式方向已经重新确认：

1. 标准商品市场
2. 定制商品市场
3. 汇率市场

这一轮的任务，是把整个市场系统从“后续继续沿一条旧路线硬推”切回到真正可持续的三分结构上，并产出下一阶段可执行的正式规格。

## 先承认当前事实

当前仓库里同时存在两类内容：

1. 正式产品方向
   - 三市场结构已经重新确认
2. 历史遗留实现与旧 prompt
   - 一度把市场错误收窄成单一标准商品撮合路线
   - 并进一步漂移成了某种材料专场语境

你不能继续沿着旧漂移方向写实现。

你需要先承认：

1. 旧的单一路线实现，只能视为一个废弃实验方向或最多是“标准商品市场的早期残片”
2. 它不能继续代表整个市场系统
3. 后续阶段必须先把三类市场的边界、职责、数据模型与入口关系写清楚，再决定具体落哪一层代码

## 这一轮的唯一目标

这一轮不追求把三类市场一次性全部做完。

这一轮只做下面四件事：

1. 把市场正式拆回三类产品线
2. 明确三类市场各自的边界、共享能力与不共享能力
3. 判断当前旧实现残留分别属于哪一类市场，哪些应冻结，哪些可复用
4. 产出接下来 1 到 2 个阶段的正式执行顺序

## 当前正式产品边界

### 1. 标准商品市场

这条线处理的是：

1. 可统一定义的商品
2. 可统一计量的商品
3. 可进入统一托管库存的商品
4. 可进入订单簿撮合的商品

这条线当前允许有：

1. 商品目录
2. 限价买单 / 限价卖单
3. 即时买入 / 即时卖出
4. 统一托管库存
5. CLAIMABLE / 异常恢复
6. 订单簿行情展示

这条线当前不等于任何单一材料专场。

也就是说：

1. 不能再把它默认等于某一类材料市场
2. 不能再把某个模组目录直接写成它的天然边界
3. 商品目录应该由制度和市场设计定义，而不是由历史实现偶然决定

### 2. 定制商品市场

这条线处理的是玩家间的非标商品交易，更接近：

1. 跳蚤市场
2. 闲鱼
3. 二手市场

这条线主要处理：

1. 单件商品
2. 带 NBT 或个体差异的商品
3. 玩家自己挂牌的商品
4. 固定价出售或后续可扩的议价机制

这条线当前不要求：

1. 统一订单簿
2. 同商品聚合撮合
3. 标准化目录先行

这条线的第一版更应该像：

1. 发布商品
2. 浏览商品
3. 查看详情
4. 买下商品
5. 下架商品
6. 待领取 / 待交付 / 留痕

### 3. 汇率市场

这条线处理的是正式货币之间的兑换关系。

它不是商品市场。

它主要处理：

1. 任务书硬币
2. 星光币
3. 其他正式货币或等价发行物

它主要关心：

1. 报价
2. 兑换规则
3. 兑换入口
4. 税费
5. 限额
6. 审计与账本

它当前不应该被混进：

1. 商品托管库存
2. 商品订单簿
3. 个体物品交易

## 三类市场共享的能力

下面这些能力可以是市场系统共享基础设施：

1. 银行结算
2. 账户冻结 / 解冻
3. 税池
4. 通知系统
5. 管理员审计入口
6. 异常恢复框架
7. MARKET 终端总入口

## 三类市场不应该混用的能力

下面这些必须拆清楚：

1. 标准商品市场的订单簿，不属于定制商品市场
2. 定制商品市场的单件挂牌，不属于标准商品市场
3. 汇率市场的兑换报价，不属于商品市场目录
4. 商品托管库存，不属于汇率市场
5. 个体物品 NBT 差异，不应被硬塞进标准商品市场

## 这一轮必须交付的内容

### A. 新的市场总设计文档

至少要产出或更新一份正式文档，明确写清：

1. 为什么市场要拆成三类
2. 三类市场各自服务什么场景
3. 三类市场分别依赖哪些底层能力
4. 三类市场分别不做什么
5. 当前旧实现残留属于哪一类市场

### B. 三类市场的最小模型拆分

至少要写清下面这些模型边界：

1. 标准商品市场
   - 商品目录
   - 托管库存
   - 订单
   - 撮合
   - CLAIMABLE
2. 定制商品市场
   - 挂牌
   - 商品快照
   - 卖家/买家状态
   - 成交与取消
   - 留痕
3. 汇率市场
   - 兑换对
   - 报价来源
   - 兑换请求
   - 审计字段
   - 限额与风控

### C. 当前旧实现的归位判断

你必须明确写出：

1. 现有旧单一路线实现哪些部分还能保留
2. 它们如果保留，应当归到标准商品市场的哪一层
3. 哪些部分应该冻结，不再继续扩写
4. 哪些旧 prompt 已经只能作为历史记录

### D. 下一阶段的正式执行顺序

你必须给出一个明确顺序，而不是泛泛而谈“以后再看”。

推荐至少写清下面这个顺序是否成立：

1. 先补三市场总设计
2. 先做汇率市场正式规则层
3. 再做标准商品市场商品目录与正式准入边界
4. 再做定制商品市场最小挂牌链
5. 最后再决定终端 MARKET 页如何拆成三类入口

如果你认为顺序要调整，也必须给出理由。

## 对当前旧实现的处理原则

如果你读到当前代码里已经存在旧的单一路线残留，处理原则应该是：

1. 先归类
2. 再判断保留或冻结
3. 不继续在错误边界上加功能

不要做成：

1. 先顺着旧实现继续堆
2. 以后再说产品边界

那样会继续跑偏。

## 给实现者的最后提醒

这一轮的关键词不是“继续做市场”，而是“先把市场重新分清楚”。

如果你做完之后仍然只有一个 MARKET，而且里面什么都想塞，那这轮就是失败的。

如果你做完之后能明确回答下面三个问题，这轮才算方向对：

1. 哪些交易属于标准商品市场？
2. 哪些交易属于定制商品市场？
3. 哪些兑换属于汇率市场？

先把这个分清楚，后面的代码才不会继续跑偏。
