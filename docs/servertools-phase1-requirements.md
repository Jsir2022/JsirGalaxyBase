# Server Tools / Cluster 第一期需求确认

日期：2026-04-11

这份文档用于记录已经确认的第一批 `ServerUtilities` 能力整合需求，供后续执行 prompt 直接引用。

## 当前结论

第一期不做 GUI，优先保证跨服工具能力先跑通。

第一期必须实现的基础命令包括：

- `home`
- `back`
- `spawn`
- `tpa`
- `rtp`
- `warp`

这些能力不能只停留在单服版，而要按后续群组服路线直接考虑跨服语义与数据库持久化。

## 功能范围确认

### 1. 必须实现的命令能力

- `home`
  - 必须实现
  - 必须具备跨服能力
- `back`
  - 必须实现
  - 必须允许跨服回退
- `spawn`
  - 必须实现
  - 定义保持不变：传送到当前 world 的出生点
  - 允许纳入跨服传送体系
- `tpa`
  - 必须实现
  - 必须允许跨服
- `rtp`
  - 必须实现
  - 后续是否所有维度都开放由制度规则控制，但能力本身必须进入一期范围
- `warp`
  - 必须实现
  - 当前定义为“传送到服务器指定地点的命名落点”
  - 需要支持跨服语义，而不是只做当前服坐标点

### 2. 跨服要求

- `home` 必须按跨服能力设计
- `warp` 必须按跨服能力设计
- `tpa` / `back` 等传送指令必须允许跨服
- 相关实现不能只依赖本地 world 存档或本地 NBT 文件

### 3. 持久化要求

- 第一期有必要直接接入数据库
- 原因：只有数据库持久化才能满足跨服状态共享要求
- 因此后续执行 prompt 不应再走“先做纯本地存储占位”的路线

### 4. 权限要求

- 当前不区分 `op` / 非 `op` 体系
- 只实现玩家权限语义
- 但权限模型后续必须允许接入：
  - 职业
  - 贡献度
  - 声望
- 因此一期权限层应保留制度扩展接口，而不是绑定 `ServerUtilities` 的旧 rank / permission handler

### 5. GUI 要求

- 第一期不做 GUI
- 原因：当前功能优先级高于 GUI
- 后续如需图形入口，可在命令主链稳定后再补

## 架构落点约束

为了满足上面的范围，第一期默认遵守下面这些工程约束：

- 相关能力应优先落在：
  - `modules.cluster`
  - `modules.servertools`
- 不直接把 `ServerUtilitiesPlayerData`、`ServerUtilitiesUniverseData`、`ServerUtilitiesNetHandler` 原样迁入
- 不接入 `ServerUtilities` 旧 GUI、旧 rank 框架或旧多通道网络注册表
- 数据持久化应走当前项目自己的数据库仓储层与 schema 管理体系

## 对 warp 的明确解释

这里的 `warp` 当前按“服务器预先定义的命名传送点”理解。

后续如果支持跨服，则一个 `warp` 目标不应只包含：

- 维度
- 坐标

还应允许扩展包含：

- 目标服务器 ID
- 目标维度
- 目标坐标
- 目标朝向或落点元数据

## 对后续 prompt 的直接影响

后续第一版执行 prompt 应默认包含：

- 以数据库为真源的跨服 `home` / `back` / `spawn` / `tpa` / `rtp` / `warp` 设计
- 玩家权限模型占位，但不引入旧 `op` 语义
- 不做 GUI，只做命令链、服务链、仓储链和必要的 cluster 接线
- 为职业 / 贡献度 / 声望后续接入预留制度判定接口

## 当前仍未展开但不再构成阻塞的问题

下面这些细节后续需要设计，但当前已经不阻塞写第一版执行 prompt：

- 单个玩家允许多少个 `home`
- `rtp` 的维度白名单与冷却策略
- `tpa` 跨服请求的超时与撤销语义
- `back` 记录最近一次传送还是多次历史
- `warp` 是否允许玩家创建，还是只允许制度/管理员维护

这些可以在执行 prompt 中明确为“先提供最小可运行实现 + 留扩展点”。
