# Terminal GUI 持续实装 Prompt（2026-04-06）

你现在接手的不是再做一套“可回退 GUI”，也不是保留新旧两套路由后让用户自己切换。

这次的要求很明确：

- 继续基于当前已经落地的终端 GUI 实现往前改
- 直接把当前 GUI 调到更接近原先预期效果
- 不要再引入 fallback GUI、legacy GUI、开关切换、双轨并存方案

如果效果还不合适，后续会由用户直接提供截图继续迭代；所以这轮应该把代码结构保持在“单实现、可继续打磨”的状态，而不是做成两套壳。

## 这轮的唯一方向

只允许做：

1. 继续修改当前 ModularUI 终端实现
2. 让当前 GUI 更接近原先想要的视觉和交互效果
3. 在不破坏银行、市场、汇率、通知、sync 主链的前提下，把页面做顺手

不允许做：

1. 新增一套旧 GUI 作为回退路径
2. 做“新旧 GUI 共存 + 配置切换”
3. 为了保险起见，把当前已经接好的 market / bank / exchange GUI 主链再退回说明页
4. 通过删功能、降级交互、静态化页面来换取“视觉稳定”

## 当前阶段已经成立的事实

这些前提不能回退：

1. 当前终端 GUI 已经基于 ModularUI，而不是原生 `GuiScreen` 手工绘制
2. 银行页、标准商品市场、定制商品市场、汇率市场都已经接入真实业务链，不再只是静态说明
3. 市场终端已经按标准商品 / 定制商品 / 汇率三条状态链拆开，不应再退回统一 MARKET 巨石页
4. 银行和市场表单已经完成 sync 语义收口，不能为了“看起来简单”再退回本地假交互
5. HUD 泡泡、确认弹窗、对象详情页这些壳层能力已经成立，应该继续用它们打磨体验，而不是删掉

## 这轮真正要解决的问题

当前不是“有没有 GUI”的问题，而是“当前 GUI 虽然能跑，但效果还不够接近最初想要的终端体验”。

这轮要处理的是：

1. 当前页面虽然已经能用，但有些区块仍偏工程态、偏拼板态
2. 有些页面虽然有对象详情节奏，但视觉和信息层次还不够像真正可长期迭代的终端
3. 有些交互虽然已经接线，但按钮、卡片、说明文本、详情区切换还不够顺手
4. 之前为排障做的一些保守处理，不应再演化成长期结构妥协

换句话说，这轮不是“先保留 fallback，以后再说”，而是直接把当前实现继续改到更像目标成品。

## 这轮的硬约束

### 1. 只维护一套 GUI 实现

要求：

1. 直接在当前 `TerminalHomeGuiFactory`、各 `PageBuilder`、`WidgetFactory` 上继续改
2. 不新增 `Legacy*Gui*`、`Fallback*Gui*`、`UseOldGui` 之类并行路径
3. 不新增“旧实现按钮”“兼容旧布局开关”“临时回退模式”

### 2. 继续实现原先效果，而不是做安全降级

这里的“原先效果”指：

1. 终端应该看起来像一个持续可用的游戏内终端，而不是调试面板
2. 玩家应该通过当前 GUI 直接完成银行、市场、汇率相关主流程，而不是被迫退回命令行
3. 页面应该保持对象优先、详情优先、动作确认优先的节奏，而不是重新退回说明板

### 3. 视觉和交互调整必须建立在现有真实业务链上

不要这么做：

1. 为了追求“原先效果”，把真实动态数据换成静态展示文案
2. 为了追求“原先效果”，临时去掉 buy / claim / exchange / transfer 这些真实动作入口
3. 为了追求“原先效果”，把详情页重新改成说明页

应该这么做：

1. 保留真实数据流
2. 保留真实动作流
3. 在真实链路上继续打磨布局、视觉层次、卡片结构、按钮节奏、留白和说明文案

## 这轮建议优先处理的 GUI 方向

### 1. 首页和导航壳继续收口成“终端”而不是“拼板”

重点：

1. 强化页头、导航、正文三层层次
2. 避免首页和总览页继续像测试卡片集合
3. 让玩家一眼知道自己当前在哪个一级页、二级页、对象详情页

### 2. 银行页继续打磨成正式业务终端

重点：

1. 让账户摘要、转账表单、公开储备、个人流水之间的层次更清晰
2. 不要再让说明块压过真实操作区
3. 确认按钮、反馈区、状态区要形成明确闭环

### 3. 标准商品市场继续做成真正的商品优先终端

重点：

1. 商品浏览区要更像浏览层，不像状态列表
2. 商品详情区要更像单个对象的交易终端
3. 盘口、动作、仓储、我的订单、claim 区要有清晰主次，不要平均铺开
4. 不要为了排版简单又退回自动选中、说明优先、全信息堆平

### 4. 定制商品市场继续做成 listing-first 的真实 GUI

重点：

1. 左侧范围 / 列表，右侧详情 / 动作的节奏要更明确
2. listing 详情应该像单件物品详情，不应再像说明面板
3. buy / cancel / claim 的可执行状态要更直观

### 5. 汇率市场继续做成标的优先详情页

重点：

1. 不要退回手持说明板
2. 标的、当前输入、正式 quote、规则字段、执行门禁要形成单页闭环
3. refresh / confirm / execute 的顺序要清晰

## 明确禁止的错误方向

不要这么做：

1. “为了避免改歪”，额外保留一套旧 GUI
2. “为了避免影响现有功能”，把当前页面只包一层旧外观壳，不真正调整当前组件结构
3. “为了稳定”，把对象详情页重新收缩成只读说明页
4. “为了更像原先效果”，引入大量静态假数据替代真实业务状态
5. “为了先交差”，做成多开关、多模式、多分支的巨石装配

## 推荐实施顺序

1. 先收口终端总壳与一级 / 二级导航层次
2. 再调整银行页与市场页的主次结构
3. 再细化标准商品、定制商品、汇率市场三类详情页的布局节奏
4. 最后补必要的回归测试和文档记录

## 验收标准

只有同时满足下面条件，这轮才算合格：

1. 没有新增 fallback GUI、legacy GUI、开关切换路径
2. 当前 GUI 仍然走真实银行 / 市场 / 汇率业务链
3. 当前 GUI 的视觉和交互更接近原先想要的终端效果，而不是更像调试板
4. 标准商品、定制商品、汇率三类子页仍保持各自边界，没有重新混回统一 MARKET 巨石页
5. 修改后至少完成一次定向构建 / 测试验证
6. `docs/WORKLOG.md` 已补记录
