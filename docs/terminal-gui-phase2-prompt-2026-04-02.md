# Terminal GUI 第二轮 Prompt：通知层、弹层与轻量视觉组件

你现在接手的是 JsirGalaxyBase 终端 GUI 的第二轮推进。本轮不是继续做纯结构收口，也不是直接跳去做完整市场 GUI。本轮目标是把已经完成第一轮排版收口的终端，继续推进到“具备通知层、弹层交互和基础视觉组件能力”的阶段，让终端真正开始向“小型操作系统壳”演进。

## 本轮前置事实

上一轮已经完成并经过人工目检确认：

1. 长文本不再明显爆框
2. 终端主内容区的排版已经比之前稳定很多
3. 目前 `TerminalHomeGuiFactory` 的主体结构已经能承担下一轮交互扩展

因此本轮不要回头再做大规模文本与卡片高度重构，除非你发现会直接妨碍本轮目标。

## 本轮目标

本轮只做下面四件事：

1. 建立可复用的终端通知层 / 轻量弹层基础设施
2. 给终端补上最小可用的图标 / 贴图 / 视觉标识能力
3. 给至少一条真实交互链补上确认弹窗或二级弹层
4. 让部分通知不必依赖“先进入主终端页”才能被玩家看到

## 本轮不做的事

1. 不直接做完整市场 GUI 功能页
2. 不重做第一页布局框架
3. 不改银行或市场业务语义
4. 不重写终端打开链
5. 不改背包入口按钮坐标系统
6. 不引入全新网络协议，除非本轮通知层必须最小新增一条客户端展示链路

## 第二轮建议范围

### 第一部分：抽出终端通知层基础设施

当前虽然已经有银行 toast overlay，但它仍然更像页面内局部提示，不是可复用的通知层。

本轮要求：

1. 抽出一层统一的终端通知模型或 helper，不要继续把通知逻辑只写死在银行页局部方法里
2. 这层通知至少要支持：
   - 标题
   - 正文
   - 级别（成功 / 提示 / 警告 / 失败）
   - 自动关闭时长
   - 可选点击动作或关闭动作
3. UI 实现可以基于：
   - `SecondaryPanel`
   - `Dialog`
   - 现有 overlay 容器
4. 目标不是做消息中心全家桶，而是先做出可复用最小能力

### 第二部分：实现“终端外可见”的轻量通知

这是本轮最重要的新能力。

要求：

1. 至少实现一种“即使玩家当前不在主终端页里，也可以看到”的轻量通知展示方式
2. 这里的“不在主终端页”不是指跨游戏系统通知，而是：
   - 玩家在普通游戏界面中
   - 或玩家只触发了某个行为，但没一直停在终端主页
3. 可以接受的形式包括：
   - 轻量弹层
   - 次级面板
   - 非模态提示窗
4. 不要求全局任意时机都弹，但至少要证明这套能力不被“必须先进入主终端”绑定死

### 第三部分：给真实交互链补一个确认弹窗

本轮至少挑一个现有真实动作，补确认弹层。

优先推荐：

1. 银行转账提交前确认
2. 开户动作确认

要求：

1. 确认弹窗使用 `Dialog` 或 `SecondaryPanel`，不要自己糊一个假确认行
2. 弹窗中至少展示关键动作摘要，例如：
   - 操作类型
   - 目标玩家
   - 金额
   - 备注摘要
3. 玩家确认后再真正触发既有业务动作
4. 玩家取消时，不得误提交

注意：

这轮是 GUI 交互层增强，不是改银行业务规则。

### 第四部分：补最小可用视觉组件

本轮至少把框架的图形能力真正用起来一次，而不只是停留在 API 已知可用。

要求：

1. 在终端首页或银行页引入至少一类真实视觉组件，优先级如下：
   - `UITexture` 图标 / 贴图
   - `ItemDisplayWidget` 物品展示
   - `ProgressWidget` 状态条
2. 这类视觉组件应服务于真实语义，不要只做装饰样板
3. 可以考虑的落点：
   - 首页状态卡的小图标
   - 银行摘要中的账户状态视觉标记
   - Exchange 公开摘要里的储备状态条

### 第五部分：给交互加基础声音钩子

本轮只要求最小落地，不做完整音频系统。

要求：

1. 至少为一个明确交互动作挂上点击音或确认音
2. 优先使用现有 `ButtonWidget.playClickSound(...)` / `clickSound(...)`
3. 如果需要更复杂声音，可最小接入 Minecraft 原生声音调用，但不要扩成一整套音频框架

## 结构要求

本轮代码应尽量沉淀成“可复用壳层”，而不是只服务一个按钮。

至少应抽出下面两类之一，最好两类都抽：

1. 终端通知 helper / builder
2. 终端 dialog / secondary panel helper

这样后续市场 GUI、职业 GUI、公共服务 GUI 都可以复用。

## 建议实施顺序

1. 先抽通知模型与基础 UI helper
2. 再把银行现有 toast 迁移或并轨到统一通知层
3. 再实现一条终端外可见的轻量通知
4. 再给银行转账或开户补确认弹窗
5. 最后补最小视觉组件和基础点击声音

## 文档要求

1. 更新 `docs/WORKLOG.md`
2. 如有必要，在 `docs/terminal-plan.md` 里补一句“终端已进入通知层与交互层阶段”
3. 若你抽出了可复用通知 / dialog 结构，可以在 `docs/` 下补一份极短说明，但不是硬性要求

## 验收要求

本轮必须同时满足下面条件才算完成：

1. 至少有一条现有真实交互链接入确认弹窗或次级弹层
2. 至少有一种通知不再被“必须停留在主终端页”绑定死
3. 至少落地一种真实图片 / 图标 / 视觉组件能力
4. 至少一个交互动作具备基础声音反馈
5. 现有终端主布局、导航和银行业务语义不被破坏

## 强制验证要求

这一条是硬性要求，不允许省略：

1. 编译出的产物需要实装到客户端和服务端
2. 客户端和服务端都需要实际启动
3. 需要进入游戏，由人工肉眼验证 GUI 的实际效果

本轮不能只用 `compileJava`、`test` 或静态阅读代码来宣称完成。

至少要人工验证下面这些点：

1. 主终端还能正常打开
2. 新增通知能实际出现
3. 终端外可见通知确实出现过，不是只在终端内部自说自话
4. 确认弹窗能拦截提交，取消不会误触发
5. 新增图标 / 视觉组件在实际游戏里没有错位或糊成一团
6. 新增声音反馈能正常触发，且不会过于扰人

## 给实现者的最后提醒

第二轮的关键词不是“把页面做更多”，而是“把终端做活”。

本轮完成后，终端应该开始具备：

- 通知能力
- 弹层能力
- 轻量视觉能力
- 可被后续市场 GUI 复用的交互壳层

不要把这轮重新做成一堆新卡片；要把它做成后续所有 GUI 页都能站上去的交互地基。
